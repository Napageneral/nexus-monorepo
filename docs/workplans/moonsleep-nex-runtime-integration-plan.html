The decision

Reuse Nex software, not Tyler's personal Nex state.
MoonSleep has a fresh Nex runtime on the Ops VPS. Nex is MoonSleep's horizontal integration and execution framework. One PostgreSQL database, separated by schemas and least-privilege roles, will store every Nex plane MoonSleep actively uses: records, search, identity, adapter state, subscriptions, schedules, jobs, queues and executor receipts. Existing MoonSleep domain systems remain the typed operational truth and are progressively fed by Nex records and jobs instead of being discarded or rebuilt.

03 · OPERATE

What we learned and resolved

The important correction was not “extract a few Nex ideas.” It was “use the tested Nex framework for the exact horizontal problems it already solves, then make it a first-class MoonSleep runtime.”

The adapter is the only provider-facing component. Records are immutable evidence. Events announce committed facts. Projectors and actions are jobs. Typed domain systems remain authoritative for their decisions and workflows.

Record and evidence contract

Keep the existing Nex record model. Make one important addition: a first-class payload object for the complete provider observation.

Record

  • Provider/platform, account or connection, record family and stable external identity.
  • Observed, provider-created and provider-updated clocks where available.
  • Sender and receiver entity/contact routing.
  • Agent-readable content summary for ordinary search.
  • Exact payload object for projectors and evidence hydration.
  • Control metadata for adapter revision, request, cursor and derived provenance.

Receipt and event

  • The ingest receipt hashes the whole committed record, including payload.
  • Same identity + same payload is idempotent; changed replay under the same immutable identity fails closed.
  • Response-page, MIME or blob custody can have a separate receipt when one response yields many records.
  • The durable record.ingested event links to the committed record and is not the raw payload itself.

Indexing rule

Payload is private and unindexed by default. Nex search indexes the safe, agent-readable record content plus approved structured facets. Authorized hydration can return payload to a projector or operator. This gives us central search without making every customer address, message body or financial row a global snippet.

Shopify payload envelope

The first adapter uses provider_object, provider_object_json and provider_object_sha256 inside payload. This preserves both lossless JSON evidence and a decoded object without polluting control metadata.

Identity and routing

Nex identity is the shared graph. MoonSleep does not invent a second party/customer hierarchy.

Entity examples

Tyler, Casey, Sean, MoonSleep, LA Pillow, Borden, a specific facility, a mailbox, the MoonSleep Shopify store, a Meta account and an installed integration can all be entities.

Contact anchors

A provider contact is anchored by platform + space_id + contact_id. For Shopify customers, the contact ID is the customer GID inside the exact shop-domain space.

Routing versus subjects

An email routes from Sean's contact to Tyler's MoonSleep mailbox contact. A Shopify order may route from the Shopify integration to MoonSleep while the customer is a separately referenced subject in payload and the typed order.

Conservative identity

No automatic merge based on name, address, email or phone alone. Ambiguity stays separate and becomes an explicit merge candidate. Canonical resolution is a reviewed operation.

The first production identity surface uses PostgreSQL-backed public Nex operations only: contacts.observe, entities.resolve, and tag list/add/remove. Direct adapter SQL is forbidden. Groups, relationships and structured entity locations are valuable follow-on primitives, but do not block the Shopify customer/order vertical.

Events, subscriptions and jobs

This is the simplification payoff: adapters ingest once, durable events fan out, and jobs stop each inventing their own polling, state and retry mechanisms.

  1. Adapter observes change. A monitor, webhook or reconciliation pull produces one or more immutable record revisions.
  2. One PostgreSQL transaction commits the spine. Record, ingest receipt, durable event, matching subscription work, queue row and dispatch receipt are stored atomically.
  3. Subscriptions decide interest. A customer projector, order projector, helpdesk context updater or alert job can subscribe to a bounded event family.
  4. The PostgreSQL executor provides delivery semantics. Idempotency, leases, retries, deadlines, terminal receipts and replay live in the common job system. Crash recovery must never require a SQLite event bridge.
  5. Projectors update typed truth. Each projector stores the source record ID, payload hash, projector revision and result/readback in its domain.
  6. Actions keep real capabilities. Provider writes use the same adapter package's explicit methods and the existing MoonSleep approval/policy boundaries.
Explicit exclusion: automated memory jobs are not part of this program. The work system is for operational adapters, projections, schedules and actions. Memory automation can be reconsidered later without blocking MoonSleep.

How the central store and typed databases coexist

Use the same PostgreSQL server on the Ops VPS and one governed MoonSleep operational database, with dedicated Nex schemas, roles and migrations beside the existing MoonSleep domain schemas. The fresh runtime boundary is defined by database roles, schema ownership, credentials and admitted MoonSleep data—not by another collection of SQLite ledgers.

Duplication is intentional when it serves two different contracts: the Nex record preserves immutable provider evidence; the MoonSleep table provides validated, indexed current state. The link is explicit and auditable, never inferred from a loose text match.

Private operator access without hiding public products

Tailscale adds a private administrative lane. It does not replace or interrupt the public storefront, Cloudflare Access, Workers or narrowly authenticated webhook ingress.

Public product plane

  • The storefront and customer-facing APIs remain publicly reachable through their existing edge and authentication boundaries.
  • The Ops web application remains behind Cloudflare Access.
  • Shopify webhooks later enter through one narrow public HMAC-verified endpoint; Nex itself is never made public for them.

Private operator plane

  • Nex continues to listen only on loopback on the Ops VPS.
  • Tyler's Mac joins the same tailnet and reaches Nex through Tailscale SSH or a bounded SSH forward.
  • Local Codex uses an explicit MoonSleep production profile isolated from Tyler's personal Nex runtime.
  • No Tailscale Funnel and no direct public Nex listener.
Timing: Tailscale does not block the PostgreSQL work. Installation and configuration can proceed autonomously; Tyler is needed only once to authorize the Mac/VPS in the chosen tailnet or provide a short-lived tagged enrollment key.

What was hashed out with Unified Ops

Core Nex lane

  • PostgreSQL operational store and migrations.
  • Record payload, receipts and durable ingest events.
  • Subscriptions, queues, jobs, schedules and idempotency.
  • PostgreSQL-backed identity public operations.
  • Runtime storage selection, search projection and fresh instance bootstrap.

Unified Ops / Shopify lane

  • Best-of-both-worlds Shopify adapter fidelity.
  • Complete customers, orders and line-item revisions.
  • Lossless provider JSON and revision hashes.
  • Resumable page staging and final manifests.
  • Identity and commerce projectors through public operations.
  • Shadow parity against Shopify and current MoonSleep tables.

Shared seam

The two lanes meet only at public runtime operations: record ingest, durable ingest events, identity operations and registered projectors. The Shopify adapter does not depend on PostgreSQL internals. Core storage contains no Shopify business rules.

Domain ownership that does not move

Dispatch still owns actual fulfillment. MoonSleep/QBO retain accounting authority. Helpdesk owns customer communication state. Provider adapter ingestion does not silently acquire domain-write or financial authority. At the same time, this plan does not remove any existing functionality: current operational paths stay live until their Nex-backed replacement proves parity and cutover.

Post-V1

Acceptance gates

Core/runtime

  • SQLite remains compatible for existing Nex deployments.
  • The MoonSleep production profile has zero active SQLite fallback.
  • PostgreSQL initialize/reapply/restart/backup/isolated-restore passes in cleanroom and production.
  • Record → receipt → event → subscription → queue is one atomic PostgreSQL transaction and is replay-safe.
  • Search, identity, adapter state, schedules, jobs, leases and executor use the selected PostgreSQL store.
  • Executor crash, lease expiry, retry, terminal failure and restart recovery are proven without duplicate work.
  • Fresh instance proves zero personal-state contamination.

Adapter

  • Exact provider payload survives without numeric or unknown-field loss.
  • Full pagination, resume, replay and reconciliation are proven.
  • Rate-limit, retry, stale cursor, tamper and hostile-link cases fail safely.
  • Existing MoonSleep data richness and provider actions do not regress.
  • Install/connect, backfill/monitor and agent-use proof all pass.

Shopify vertical

  • Each customer GID maps to at most one active Shopify contact anchor.
  • Checkpointed projection twice produces no duplicate records, entities, contacts, observations, orders or jobs.
  • Every production batch is capped, single-worker, health-gated and pauses before the global resource guard must shed critical work.
  • Customer updates create a record revision while preserving entity binding.
  • Orders and line items link to the correct entity and preserve immutable billing/shipping snapshots.
  • Counts, hashes and typed totals reconcile with source/current MoonSleep.
  • Reconciliation pulls recover missed or out-of-order webhooks.

Access and cutover

  • Nex remains loopback-only and private operator access works from an isolated Mac profile over Tailscale SSH.
  • Storefront, Cloudflare Access Ops UI and public customer APIs remain independently reachable.
  • Webhook ingress is narrow, HMAC-verified and cannot expose the Nex operator surface.
  • Old functionality remains live throughout shadowing.
  • Rollback is tested before a duplicate path is retired.
  • Dispatch, accounting and other domain ownership boundaries remain intact.

Explicit non-goals

  • No import of Tyler's or Casey's private/personal Nex corpus.
  • No revival of the repository-corpus/search substrate or dedicated substrate service.
  • No automated memory jobs in this program.
  • No active SQLite ledger for any operation used by the MoonSleep production profile.
  • No public Nex listener, no Tailscale Funnel and no storefront dependency on Tailscale.
  • No universal replacement for typed MoonSleep domain tables.
  • No direct provider adapter writes into domain tables.
  • No forced generic commerce abstraction before Shopify fidelity is proven.
  • No product/BOM/manufacturing expansion, partner portal, TikTok Shop or Amazon ingestion until Shopify V1 is terminal.
  • No schema field with the prohibited generic discriminator name.
  • No removal of a working capability merely because its implementation has not yet migrated to Nex.
